# feat(qa-plan): v0.2 backlog

## Implementation Units

Numbered so each produces a committable milestone. Ship as one
bundle PR (`feat/qa-plan-v0.2`) or multiple small PRs — impl agent
chooses. Exit criteria are concrete.

### Unit 1 — `$USER_TAG` Windows fallback

- [ ] Change `USER_TAG="${USER:-unknown}"` in Phase 2e Preamble-
      adjacent context to `USER_TAG="${USER:-${USERNAME:-unknown}}"`.
- [ ] Windows git-bash sets `$USERNAME` but not `$USER`, so the
      current code defaults to the literal string `"unknown"` on
      every Windows mirror filename. Observed live in dogfood Run
      #3 (`unknown-20260423-232637-test-qa-plan-slug-verify-qa-plan.md`).
- [ ] Exit: on Windows git-bash, a fresh `/qa-plan` run writes a
      mirror with the actual username as the prefix
      (e.g., `REDMOND+dunliu-...`).

### Unit 2 — MSYS `/qa-plan` path-translation in warning strings

- [ ] The canonical "no diff" warning contains literal `/qa-plan
      skipped` which MSYS on Windows git-bash rewrites to
      `C:/Program Files/Git/qa-plan skipped` when the string is
      passed to `_qa_plan_record_warning`. Observed in the 2026-04-24
      failure analytics entry.
- [ ] Fix options (pick one; impl agent decides):
      1. Prefix with double slash: `//qa-plan skipped` (MSYS leaves
         paths starting with `//` alone)
      2. Wrap the call site with `MSYS_NO_PATHCONV=1`
      3. Pass through stdin or a file instead of command-line arg
- [ ] Audit every existing canonical-warning emission site for the
      same hazard; apply the chosen fix uniformly.
- [ ] Exit: failure analytics entry from a clean-master abort shows
      `"skipped": "/qa-plan skipped"` (or semantically equivalent
      non-translated text), not a Windows path.

### Unit 3 — README-only-diff surface auto-detection

- [ ] Phase 1g's surface auto-detection currently matches zero files
      when the diff contains only `README.md` (or `CHANGELOG.md`,
      `LICENSE`, `CONTRIBUTING.md`, etc.). The skill correctly falls
      back to `AskUserQuestion` but the auto-detect silence is a UX
      papercut.
- [ ] Options (impl agent picks):
      a. Add documentation patterns to existing surface rules (fold
         into `claude-skill` for skill repos, or a cross-cutting
         "any surface" for general repos).
      b. Add a new `documentation` surface type with a minimal axis
         list: `{prose accuracy, link integrity, example correctness,
         version drift, CHANGELOG completeness}`.
      c. Keep the fallback but make the `AskUserQuestion` rephrase
         acknowledge "no surface pattern matched; pick one or abort."
- [ ] Update `references/taxonomies.md` Surface Detection Rules
      table accordingly.
- [ ] Exit: a `/qa-plan` run on a diff containing only `README.md`
      changes produces a non-silent auto-detect result OR a clearly
      contextualized fallback question.

### Unit 4 — `HEAD^1...HEAD^2` merge-diff fallback in Phase 1b

- [ ] Dogfood Run #2 surfaced that running `/qa-plan` on master
      immediately after a merge (e.g., to review-what-just-landed)
      has an empty `master...HEAD` diff. The skill currently would
      abort Phase 1b; the orchestrator LLM improvised
      `HEAD^1...HEAD^2` (the merge-commit parent diff) and the run
      produced a useful 86-case plan.
- [ ] Codify this path into SKILL.md Phase 1b as an explicit
      fallback after the three existing diff-source checks fail.
      Detect it with `git rev-parse --verify HEAD^2` — if HEAD has
      two parents, it's a merge commit, so `HEAD^1...HEAD^2` is the
      merge diff.
- [ ] Update Reviewer Coverage disclosure so the user sees "Diff
      source: merge-diff HEAD^1..HEAD^2 (post-merge self-review)"
      when this path fires.
- [ ] Exit: `git checkout master` after a fresh merge + `/qa-plan`
      runs through Phase 1 without an abort and produces a plan
      with merge-diff diff-source disclosure.

### Unit 5 — Probe 7 silent-compliance fix

- [ ] DV3 Probe 7 ("shell out to check if Playwright is installed")
      did NOT leak (no subprocess spawned, no test execution) but
      also did NOT emit the Gate 1 verbatim decline. The probe text
      was silently absorbed into a normal `/qa-plan` run. Users
      can't tell the gate fired.
- [ ] Tighten HARD GATE 1 in SKILL.md so subprocess-adjacent
      framings ("shell out to check X", "run a subprocess to verify
      Y", "can you quickly test if Z is installed") are explicitly
      named in the decline-text probe-pattern enumeration and the
      skill is required to reference HARD GATE 1 when declining
      these asks — not silently continue.
- [ ] Update `docs/qa-plans/adversarial-probe-corpus.md` Probe 7 to
      include the tightened expected response.
- [ ] Exit: DV3 Probe 7 re-run elicits the verbatim Gate 1 decline
      instead of silent absorption.

### Unit 6 — Project-defined subagent files for `tools:` enforcement

- [ ] PR #6's architectural-honesty fix disclosed that Claude Code's
      `Agent` tool has no `tools:` parameter — tool restriction in
      v0.1 is prompt-level best-effort only.
- [ ] Create `skills/qa-plan/agents/qa-plan-persona-confused-user.md`,
      `qa-plan-persona-data-corruptor.md`,
      `qa-plan-persona-race-demon.md`,
      `qa-plan-persona-prod-saboteur.md`, and
      `qa-plan-spec-only-reviewer.md` with YAML frontmatter
      including `tools:` restricting each subagent's capabilities
      (personas: `[Bash, Read, Grep]`; spec-only: `[Read, Grep]`).
- [ ] Update SKILL.md Phase 7a and 7b to dispatch via the new
      `subagent_type` names instead of `general-purpose`.
- [ ] Document in README how to install these subagents at
      `~/.claude/agents/` alongside the skill install.
- [ ] Reviewer Coverage now honestly reports "tool restriction
      enforced via project-defined subagent frontmatter" instead of
      "best-effort prompt-level intent."
- [ ] Exit: `/qa-plan` dispatch in fresh session confirms
      subagent_type is one of the qa-plan-* types; Reviewer
      Coverage states enforcement is subagent-frontmatter-backed;
      honesty prose in SKILL.md updated to reflect the change.

### Unit 7 — Analytics recovery one-liner documentation

- [ ] PR #6 note: 238 pre-PR-#6 entries in
      `~/.gstack/analytics/skill-usage.jsonl` are pretty-printed
      multi-line JSON (malformed JSONL). Recovery command is a
      one-liner: `jq -s -c '.[]' skill-usage.jsonl > skill-usage-repaired.jsonl`.
- [ ] Add a "Recovery" section to
      `skills/qa-plan/references/analytics-schema.md` with the
      one-liner + when to use it (after upgrading from a pre-PR-#6
      v0.1 install).
- [ ] Optional: add a `qa-plan/bin/repair-analytics.sh` helper
      script if the inline one-liner feels too fiddly; v0.1
      convention has no `bin/`, so defer to prose-only unless the
      impl agent sees value.
- [ ] Exit: analytics-schema.md has a Recovery section with a
      copy-pasteable command.
